When we started building Svantic, we faced a fundamental question: how should agents work together?

The naive approach is hardcoded integrations. Agent A calls Agent B. Agent B calls Agent C. Wire them together in code.

This works for demos. It fails at scale. What happens when you have 50 agents? 200? What happens when agents are added and removed dynamically?

We needed a protocol for agent collaboration. We call it A2A — Agent-to-Agent.

## The Problem

Traditional multi-agent systems have two approaches:

**1. Centralized Orchestration**

A "manager" agent controls everything:

```
Manager → "Agent A, do X"
Agent A → result
Manager → "Agent B, do Y with result"
Agent B → result
Manager → "Done"
```

This is easy to reason about but creates bottlenecks. The manager becomes a single point of failure. The manager needs to know about every agent's capabilities upfront.

**2. Hardcoded Pipelines**

Agents are wired together in code:

```python
result = agent_a.process(input)
result = agent_b.transform(result)
result = agent_c.validate(result)
return result
```

This is fast but inflexible. Adding a new agent requires code changes. The system can't adapt to runtime conditions.

## A2A: Dynamic Discovery and Collaboration

A2A is a protocol that lets agents discover each other and collaborate dynamically. No central manager. No hardcoded integrations.

### Agent Cards

Every agent publishes an Agent Card — a machine-readable description of its capabilities:

```yaml
name: invoice-processor
description: Extracts structured data from invoice documents
capabilities:
  - name: extract_fields
    description: Extract vendor, amount, date, line items from invoice
    input_schema:
      type: object
      properties:
        document_url: { type: string }
        document_type: { type: string, enum: [pdf, image, email] }
    output_schema:
      type: object
      properties:
        vendor: { type: string }
        amount: { type: number }
        date: { type: string, format: date }
        line_items: { type: array }
protocols:
  - a2a/1.0
  - mcp/1.0
authentication:
  type: oauth2
  scopes: [documents:read]
```

Agent Cards are registered with the Mesh and discoverable by any other agent.

### Capability Discovery

When an agent needs help, it queries the Mesh:

```typescript
const agents = await mesh.discover({
  capability: 'extract_fields',
  input: { document_type: 'invoice' },
  constraints: { latency_ms: 2000, cost_budget: 0.10 }
});

// Returns:
[
  { 
    agent: 'invoice-processor', 
    capability: 'extract_fields',
    estimated_latency_ms: 1200,
    estimated_cost: 0.03,
    success_rate: 0.97
  },
  {
    agent: 'generic-document-agent',
    capability: 'extract_structured_data',
    estimated_latency_ms: 1800,
    estimated_cost: 0.05,
    success_rate: 0.89
  }
]
```

The requesting agent can now choose the best collaborator based on its criteria.

### Task Delegation

Once an agent is selected, the requesting agent sends a task:

```typescript
const result = await mesh.delegate({
  agent: 'invoice-processor',
  capability: 'extract_fields',
  input: {
    document_url: 'https://storage.example.com/invoices/inv-123.pdf',
    document_type: 'pdf'
  },
  timeout_ms: 5000,
  callback_url: 'https://my-agent.example.com/callbacks'
});
```

The Mesh routes the request to the target agent, handles retries if needed, and returns the result.

### Emergent Workflows

The power of A2A is that workflows emerge from capabilities, not code.

Consider a task: "Process this invoice and update the accounting system."

With A2A:

1. The initiating agent queries for `extract_fields` capability → finds `invoice-processor`
2. Delegates extraction → receives structured invoice data
3. Queries for `update_accounting_entry` capability → finds `accounting-agent`
4. Delegates update → receives confirmation
5. Task complete

No hardcoded pipeline. If tomorrow you deploy a better invoice processor, the workflow automatically uses it. If the accounting system is down, the Mesh handles retries.

## Protocol Spec

A2A messages follow a simple structure:

```typescript
interface A2ARequest {
  protocol: "a2a/1.0";
  message_type: "task" | "response" | "error" | "heartbeat";
  request_id: string;
  source_agent: string;
  target_agent: string;
  capability: string;
  input: Record<string, unknown>;
  metadata: {
    trace_id: string;
    timeout_ms: number;
    priority: "low" | "normal" | "high";
  };
}

interface A2AResponse {
  protocol: "a2a/1.0";
  message_type: "response";
  request_id: string;
  source_agent: string;
  output: Record<string, unknown>;
  metadata: {
    execution_time_ms: number;
    tokens_used?: number;
  };
}
```

Transport is WebSocket for real-time communication, with HTTP fallback for request/response patterns.

## Security

A2A includes security by default:

**Authentication**: Agents authenticate with the Mesh using API keys or mTLS. The Mesh verifies identity before routing requests.

**Authorization**: Agent Cards specify required scopes. The Mesh enforces that requesting agents have appropriate permissions.

**Encryption**: All A2A traffic is encrypted in transit (TLS 1.3).

**Audit**: Every A2A interaction is logged with full trace context.

## Comparison to MCP

You might wonder how A2A relates to Anthropic's Model Context Protocol (MCP).

They're complementary:

- **MCP** is for LLM ↔ Tool communication (how an LLM calls a function)
- **A2A** is for Agent ↔ Agent communication (how agents collaborate)

Svantic supports both. Your agents can expose MCP-compatible tools for direct LLM access, and A2A capabilities for agent collaboration.

```yaml
protocols:
  - a2a/1.0  # Agent-to-agent
  - mcp/1.0  # LLM-to-tool
```

## Get Started

A2A is built into the Svantic SDK. To make your agent discoverable:

```typescript
const agent = new Agent({
  name: 'my-agent',
  mesh: { url: 'wss://mesh.svantic.com' }
});

agent.define_capability({
  name: 'my_capability',
  description: 'Does something useful',
  schema: { /* JSON Schema */ },
  handler: async (input) => {
    // Your logic here
    return result;
  }
});

await agent.start();
```

Your agent is now discoverable and can participate in dynamic workflows.
