When we started building Svantic, we faced a fundamental question: how should agents work together?

The naive approach is hardcoded integrations. Agent A calls Agent B. Agent B calls Agent C. Wire them together in code.

This works for demos. It fails at scale. What happens when you have 50 agents? 200? What happens when agents are added and removed dynamically?

We needed a protocol for agent collaboration. We adopted A2A — Agent-to-Agent — the open standard for agent interoperability.

## The Problem

Traditional multi-agent systems have two approaches:

**Centralized Orchestration** — A "manager" agent controls everything. Easy to reason about but creates bottlenecks. The manager becomes a single point of failure and needs to know about every agent's capabilities upfront.

**Hardcoded Pipelines** — Agents are wired together in code. Fast but inflexible. Adding a new agent requires code changes. The system can't adapt to runtime conditions.

## A2A: Dynamic Discovery and Collaboration

A2A is a protocol that lets agents discover each other and collaborate dynamically. No central manager. No hardcoded integrations.

### Agent Cards

Every agent publishes an Agent Card — a machine-readable description of its capabilities:

```json
{
  "name": "invoice-processor",
  "description": "Extracts structured data from invoice documents",
  "url": "http://invoice-processor:4000/send",
  "version": "1.0.0",
  "capabilities": { "streaming": true },
  "skills": [
    {
      "id": "extract_fields",
      "name": "extract_fields",
      "description": "Extract vendor, amount, date, line items from invoice",
      "parameters": {
        "type": "object",
        "properties": {
          "document_url": { "type": "string" },
          "document_type": { "type": "string", "enum": ["pdf", "image", "email"] }
        }
      }
    }
  ]
}
```

Agent Cards are published at `/.well-known/agent-card.json` and registered with the Mesh. Any agent can discover them.

### Capability Discovery

When an agent needs help, it queries the Mesh:

```typescript
const agents = await mesh.discover({
  capability: 'extract_fields',
  input: { document_type: 'invoice' },
  constraints: { latency_ms: 2000, cost_budget: 0.10 }
});

// Returns matching agents ranked by fit
```

The requesting agent can choose the best collaborator based on latency, cost, success rate, or custom criteria.

### Task Dispatch

Once an agent is selected, the requesting agent sends a task via JSON-RPC 2.0:

```typescript
const result = await mesh.delegate({
  agent: 'invoice-processor',
  capability: 'extract_fields',
  input: {
    document_url: 'https://storage.example.com/invoices/inv-123.pdf',
    document_type: 'pdf'
  },
  timeout_ms: 5000
});
```

The Mesh routes the request, handles retries if needed, and returns the result.

## Emergent Workflows

The power of A2A is that workflows emerge from capabilities, not code.

Consider a task: "Process this invoice and update the accounting system."

With A2A:

- The initiating agent queries for `extract_fields` capability → finds `invoice-processor`
- Delegates extraction → receives structured invoice data
- Queries for `update_accounting_entry` capability → finds `accounting-agent`
- Delegates update → receives confirmation
- Task complete

No hardcoded pipeline. If tomorrow you deploy a better invoice processor, the workflow automatically uses it. If the accounting system is down, the Mesh handles retries.

## Registration Flow

Remote agents register with the Mesh in two steps:

```
1. POST /auth/get_token { client_id, client_secret, agent_type }
   → { token: "eyJ..." }

2. POST /agents/register (Bearer token)
   { agent_type, instance_id, url, tools, mcp_servers }
   → Mesh fetches agent card, creates capability proxies
   → { ok: true, tenant_id }
```

From that moment, the agent's capabilities are discoverable and invocable by any other agent in the same tenant.

## Streaming

For long-running tasks, A2A SSE streaming provides real-time progress:

```
POST /send (message/stream)
  → SSE: TaskStatusUpdateEvent (working...)
  → SSE: TaskStatusUpdateEvent (working...)
  → SSE: TaskArtifactUpdateEvent (result)
  → SSE: TaskStatusUpdateEvent (completed)
```

Clients can display progress, intermediate results, and handle long-running operations gracefully.

## Security

A2A includes security by default:

**Authentication** — Agents authenticate with the Mesh using API keys or mTLS. The Mesh verifies identity before routing requests.

**Multi-Tenant Isolation** — Tenant A's agents are invisible to Tenant B. Cryptographic boundary.

**Authorization** — Agent Cards specify required scopes. The Mesh enforces permissions.

**Encryption** — All A2A traffic is encrypted in transit (TLS 1.3).

**Audit** — Every A2A interaction is logged with full trace context.

## Relationship to MCP

You might wonder how A2A relates to Anthropic's Model Context Protocol (MCP).

They're complementary:

- **MCP** is for LLM ↔ Tool communication (how an LLM calls a function)
- **A2A** is for Agent ↔ Agent communication (how agents collaborate)

Svantic supports both. Your agents can expose MCP-compatible tools for direct LLM access, and A2A capabilities for agent collaboration:

```typescript
// Register an MCP server — its tools become A2A capabilities
await agent.register_mcp('filesystem', {
  command: 'npx',
  args: ['-y', '@modelcontextprotocol/server-filesystem', '/workspace'],
});
```

## Get Started

A2A is built into the Svantic SDK. To make your agent discoverable:

```typescript
const agent = new Agent({
  name: 'my-agent',
  description: 'Does something useful',
  url: 'http://localhost:4000',
});

agent.define_capability({
  name: 'my_capability',
  description: 'What this capability does',
  parameters: { /* JSON Schema */ },
  handler: async (args) => {
    return { result: 'done' };
  }
});

agent.expose(app);
await mesh.connect();
```

Your agent is now discoverable and can participate in dynamic workflows.
