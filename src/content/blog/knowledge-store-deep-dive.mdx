Most AI agents have amnesia. Every session starts from scratch. Every mistake is repeated. Every successful strategy is forgotten.

Svantic's Knowledge Store changes that. Here's how we built institutional memory for AI agents.

## The Problem with Stateless Agents

Consider a document processing workflow. The first time an agent encounters an invoice from Acme Corp, it might:

- Try to extract fields using a generic template
- Fail because Acme uses a non-standard layout
- Fall back to OCR + LLM extraction
- Eventually succeed after 3-4 attempts

The second time it sees an Acme invoice? Same process. Same failures. Same wasted tokens.

Multiply this by thousands of document types and millions of executions. The cost compounds. The latency compounds. The unreliability compounds.

## The Learning Cycle

The Knowledge Store observes every execution and extracts reusable patterns:

```
Execution #1: Acme invoice processed
├── Attempts: 4
├── Successful strategy: OCR → field mapping → validation
├── Layout signature: hash(structure)
└── Indexed for future reference

Execution #2: Acme invoice (different document)
├── Knowledge Store: "I've seen this layout before"
├── Strategy retrieved: OCR → field mapping → validation
├── Attempts: 1
└── 75% faster, 60% fewer tokens
```

The 50,000th Acme invoice is processed as efficiently as if we'd written custom code for it.

## Architecture

The Knowledge Store has three layers:

### Observation Layer

Every agent execution emits structured events — task type, attempts, strategies tried, success/failure, input characteristics. These events are captured automatically with no instrumentation required.

### Indexing Layer

Raw events are processed into retrievable knowledge:

**Pattern Extraction**: Successful strategies are indexed by input characteristics. A document layout hash maps to the extraction strategy that worked.

**Failure Analysis**: Failed attempts are analyzed to avoid future mistakes. If template matching always fails for Acme invoices, the system learns to skip it.

**Embedding Generation**: Unstructured inputs are embedded for semantic retrieval. "Process this insurance claim" retrieves knowledge from similar past tasks.

### Retrieval Layer

Before executing a task, agents query the Knowledge Store:

```typescript
const knowledge = await knowledge_store.query({
  task: 'extract_invoice_fields',
  input_signature: document_hash,
  context: { document_type: 'invoice' }
});

// Returns:
{
  suggested_strategy: 'ocr_extraction',
  confidence: 0.95,
  similar_executions: 847,
  average_duration_ms: 890,
  known_pitfalls: ['date_format_varies', 'address_sometimes_missing']
}
```

The agent uses this knowledge to skip failed strategies and apply proven approaches.

## The Compounding Effect

Knowledge compounds over time:

| Stage | Knowledge Level | First-try Success | Human Help |
|---|---|---|---|
| **100 tasks** | Low | 20% | High |
| **1,000 tasks** | Medium | 65% | Medium |
| **10,000 tasks** | High | 90%+ | Rare |

Each execution feeds the store. Knowledge from one domain helps others. The scraper learned that insurance portals have a "My Documents" tab? That pattern helps when the AI encounters a similar portal from a different carrier.

## Staleness Detection

Knowledge can go stale. Acme Corp might redesign their invoices. A strategy that worked last month might fail today.

The Knowledge Store tracks success rates over time:

```
strategy: ocr_extraction + layout: acme_standard_v2
├── Last 7 days: 98% success (n=234)
├── Last 30 days: 97% success (n=1,847)
└── Status: HEALTHY

strategy: template_match + layout: vendor_x_2024
├── Last 7 days: 45% success (n=89)
├── Last 30 days: 82% success (n=412)
└── Status: DEGRADED — re-learning triggered
```

When success rates drop, the system flags the pattern as stale, reduces confidence scores, triggers re-learning from recent executions, and alerts operators if degradation persists.

## Central + Sidecar: Fleet Learning

In standalone or sidecar deployments, knowledge is local to each instance. But with Central + Sidecar (Hub Mesh) topology:

- **Sidecars contribute knowledge upward** — their learnings benefit the whole fleet
- **Sidecars retrieve knowledge from central** — they benefit from other sidecars' learnings

What Pod A learned on Monday is available to Pod C on Tuesday. This creates fleet-wide institutional intelligence.

## Privacy and Isolation

Knowledge is scoped by tenant. Your patterns never leak to other customers.

For sensitive data, the Knowledge Store supports:

- **Input hashing** — Only signatures are stored, not raw inputs
- **Output redaction** — Configurable rules to strip PII before indexing
- **Retention policies** — Automatic expiration of old knowledge
- **Encryption at rest** — AES-256 for all stored data
