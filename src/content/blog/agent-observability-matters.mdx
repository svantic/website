You've deployed AI agents to production. They're processing real workloads. Something goes wrong.

How do you debug it?

If you've tried using traditional observability tools (Datadog, New Relic, Prometheus) for AI agents, you've felt the pain. These tools are great for deterministic systems. Agents are fundamentally different.

## What Makes Agents Different

### Non-Deterministic Behavior

Traditional services: same input → same output.

Agents: same input → potentially different outputs, different tool calls, different reasoning paths.

When an agent fails, you can't just replay the request. You need to understand the entire decision chain.

### Multi-Turn Interactions

A REST API call is a single request/response. An agent session might involve:

- 15 LLM calls
- 8 tool invocations  
- 3 sub-agent delegations
- 40+ internal state transitions

The "trace" isn't a simple waterfall — it's a tree.

### Hidden Reasoning

The most important information is often invisible. Why did the agent choose tool A over tool B? Why did it retry? What context influenced the decision?

Traditional traces show what happened. Agent traces need to show **why**.

### Cost Attribution

Agent operations have variable costs. An LLM call might cost $0.001 or $0.10 depending on context size. A workflow might make 5 LLM calls or 50.

You need cost visibility at every level: per-session, per-agent, per-capability, per-customer.

## Svantic's Approach

Svantic's telemetry was designed for agents from day one.

### Zero-Config by Default

Telemetry is enabled automatically. Every agent records spans, LLM usage, and events with no setup required:

```typescript
const agent = new Agent({
  name: 'ticket-agent',
  description: 'Manages support tickets.',
  url: 'http://localhost:4000',

// That's it. Telemetry is already working.
```

When connected to the Mesh, telemetry exports to the Svantic dashboard. In standalone mode, it exports to the console.

### Hierarchical Traces

Every session generates a trace tree:

```
Session: "Process invoice batch"
├── Agent: invoice-processor
│   ├── LLM Call: classify_documents (1,200 tokens, 890ms)
│   ├── Tool Call: download_file (45ms)
│   ├── LLM Call: extract_fields (3,400 tokens, 1,240ms)
│   └── Sub-Agent: validation-agent
│       ├── Tool Call: lookup_vendor (120ms)
│       └── LLM Call: validate_data (800 tokens, 450ms)
├── Agent: routing-agent
│   └── LLM Call: determine_route (600 tokens, 320ms)
└── Summary: 6 operations, 6,000 tokens, 3.1s total
```

Drill down from session to agent to individual operation.

### Span Types

Different operations need different instrumentation:

**Generic spans** for any operation:

```typescript
const span = agent.tracer.start_span('operation_name');
span.set_attribute('custom.attribute', 'value');
span.end();
```

**Tool spans** with the tool name as an automatic attribute:

```typescript
const span = agent.tracer.start_tool_span('get_ticket');
span.set_attribute('ticket.id', 4521);
span.end();
```

**LLM spans** that capture model and token usage:

```typescript
const span = agent.tracer.start_llm_span('gemini-2.0-flash');
// ... make LLM call ...
span.end({ input_tokens: 150, output_tokens: 42 });
```

### W3C Trace Propagation

When your agent calls another agent (via A2A), trace context propagates automatically using the W3C `traceparent` header. This creates unified traces across agent-to-agent calls.

For non-A2A HTTP calls, inject context manually:

```typescript
import { ContextInjector } from '@svantic/sdk/telemetry';

const headers = { 'Content-Type': 'application/json' };
const context = agent.tracer.get_current_context();
if (context) {
  ContextInjector.inject(headers, context);
}

await fetch('https://external-service.com/api', { headers });
```

### OpenTelemetry Export

Svantic exports traces in OpenTelemetry format. Send to your existing stack:

```typescript
import { SvanticTelemetry } from '@svantic/sdk';

const telemetry = SvanticTelemetry.init({
  service_name: 'ticket-agent',
  export: {
    endpoint: 'http://otel-collector:4318/v1/traces',
    protocol: 'otlp',
  },
});

const agent = new Agent({
  name: 'ticket-agent',
  telemetry,
});
```

Your Datadog, Grafana, or Honeycomb dashboards can include agent traces alongside infrastructure metrics.

## Practical Debugging

Here's a real debugging session:

**Symptom**: Customer reports invoice processing is slow.

**Step 1**: Check the dashboard. P95 latency is 8.2s (normally 2.1s).

**Step 2**: Drill into recent traces. Find one that took 9.1s.

**Step 3**: Expand the trace tree:
- `classify_documents`: 890ms ✓
- `extract_fields`: 6,200ms ✗ (usually 1,200ms)
- `validate_data`: 450ms ✓

**Step 4**: Inspect `extract_fields`:
- Input: 12-page document (usually 2-3 pages)
- LLM calls: 5 (usually 1)
- Token usage: 34,000 (usually 3,000)

**Step 5**: Root cause identified — large document triggered multiple extraction passes.

**Step 6**: Fix — add chunking strategy for documents over 5 pages.

Total debug time: 4 minutes.

## Events

For discrete signals outside the span tree:

```typescript
agent.telemetry.event('ticket.escalated', {
  ticket_id: 4521,
  reason: 'SLA breach',
});
```

Events are buffered and flushed alongside spans. If emitted within an active span, they capture `trace_id` and `span_id` for correlation.

## Dashboard Integration

Traces appear in the Svantic dashboard under the session timeline:

## Retention and Compliance

Trace data can be sensitive. Svantic supports:

- **Configurable retention** — 7 days, 30 days, or custom
- **PII redaction** — Automatic or rule-based redaction before storage
- **Export controls** — Restrict who can export raw traces
- **Compliance modes** — HIPAA, SOC 2, GDPR configurations

Observability shouldn't compromise security.
